大型跨境电商后端所有核心模块之间的架构关系、调用流向、依赖层次、数据流转。
不画虚图，直接讲 “谁依赖谁、谁调用谁、谁被谁调用”。
一、整体后端分层结构（从上到下）
接入层：API 网关、路由、鉴权、限流
BFF 层（Backend For Frontend）：多端聚合接口
业务微服务层（核心）
基础能力中台层：通用服务，不感知业务
数据层：DB、缓存、消息队列、搜索引擎
第三方集成层：支付、物流、海关、短信、海外渠道
所有模块只允许上层调用下层，禁止反向依赖。
二、核心业务模块划分（后端本体）
1. 用户域
用户中心服务（user-service）
会员 / 积分服务（member-service）
账号安全 / 风控（risk-account）
作用：提供账号、登录、信息、权限。
被谁依赖：几乎所有服务都依赖它（订单、购物车、支付、客服…）
2. 商品域
商品服务（product-service）
类目 / 品牌服务（category-service）
库存服务（inventory-service）
价格服务（price-service）
商品搜索服务（search-service）
关系：
product 提供 SPU/SKU
price 负责多币种、汇率、促销价
inventory 负责真实库存、预占、锁定
search 依赖 product 同步数据
被谁依赖：购物车、订单、结算、推荐、活动
3. 交易核心域（最复杂、最关键）
3.1 购物车服务（cart-service）
依赖：user、product、price、inventory
只做暂存，不产生交易
3.2 订单服务（order-service）
整个后端的核心枢纽
强依赖：user、product、price、inventory、payment
弱依赖：logistics、customs、after-sale
订单是所有下游服务的 “源头”：
订单创建 → 触发支付 → 触发清关 → 触发物流 → 触发售后
3.3 交易促销服务（promotion-service）
优惠券、满减、秒杀、团购、渠道价
被 cart、order 调用
依赖 price、product
4. 支付财务域
4.1 支付服务（payment-service）
对接 PayPal、Stripe、信用卡、本地钱包
被 order 调用
回调后反过来通知 order 状态变更
4.2 结算服务（settlement-service）
平台与商家分账、跨境结算、结汇
依赖 order、payment、merchant
4.3 账务 / 账单服务（bill-service）
依赖订单、支付、商家佣金
4.4 汇率服务（exchange-service）
全局唯一汇率来源
被 price、order、payment、settlement 共同依赖
5. 跨境履约域（跨境电商特色）
5.1 物流服务（logistics-service）
海外仓、尾程配送、面单、轨迹
依赖 order、user（地址）
订单支付成功后，由 order 触发物流单创建
5.2 清关服务（customs-service）
三单对碰、海关申报、HS 编码、税费
依赖 order、payment、logistics
必须在发货前完成
5.3 关务税金服务（tax-service）
VAT/GST/ 关税计算
被 order、price 调用
6. 商家与平台运营域
6.1 商家服务（merchant-service）
店铺、入驻、商品发布权限、结算规则
被 product、order、settlement 依赖
6.2 平台运营服务（operation-service）
活动、频道、Banner、榜单
依赖 product、order
7. 售后与客服域
售后服务（after-sale-service）
客服工单（ticket-service）
跨境纠纷处理
依赖关系：
依赖 order、payment、logistics
可反向触发退款、退货、重发
8. 基础中台模块（被所有业务依赖）
消息队列服务（消息异步解耦）
配置中心
分布式锁 / 幂等服务
国际化服务（i18n-service）
地区 / 口岸规则服务（region-service）
文件 / 图片服务（oss-service）
短信 / 邮件 / Push（message-service）
三、模块之间最核心的调用关系（从上到下）
1. 下单主链路（最关键）
plaintext
网关 → BFF → cart
→ order（创建订单）
   ↓ 调用库存预占
   inventory（预扣库存）
   ↓ 调用价格+促销
   price + promotion
   ↓ 调用支付
   payment
   ↓ 支付成功回调
   order（更新待发货）
   ↓ 触发清关
   customs
   ↓ 清关完成
   logistics（发货）
   ↓ 物流签收
   order（完成）
2. 数据流向（非常重要）
商品数据 → 搜索 / 推荐 / 购物车 / 订单
订单数据 → 支付 / 物流 / 清关 / 售后 / 结算 / 大数据
支付数据 → 订单 / 财务 / 风控
用户数据 → 全链路
3. 异步解耦关系（Kafka/RocketMQ）
为了避免服务雪崩，跨境电商大量用消息：
订单创建 → 发送消息给支付、物流、清关
支付成功 → 发送消息通知订单、风控、运营
物流发货 → 发送消息通知订单、会员、消息推送
库存变化 → 发送消息同步搜索、购物车
微服务之间尽量异步，避免长链路同步调用。
四、依赖禁止规则（架构红线）
下游服务绝不允许调用上游服务
例：物流不能主动调订单；支付不能主动调购物车
核心域不依赖非核心域
订单、支付、库存不依赖运营、活动、客服
基础中台不依赖任何业务服务
跨境域（清关 / 物流）不反过来影响交易主流程
清关失败只打标记，不回滚订单链路
五、一句话总结后端模块架构关系
用户、商品是基础数据层；
订单是整个后端的中枢枢纽；
支付、物流、清关围绕订单展开；
促销、商家、售后是交易扩展；
所有模块通过消息队列异步解耦，
通过基础中台统一能力，
形成 “上游依赖下游、同步执行业务、异步完成履约” 的稳定跨境电商后端架构。
